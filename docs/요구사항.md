## REQ-CONSULTING-002 회의록 분석 탭

### 회차별 노트 관리

```javascript
note = {
  id: string,
  seq: number,           // 1차, 2차 ...
  title: string,
  date: ISO date,
  content: string,       // 회의록 본문
  analysis: string,      // AI 분석 결과 (마크다운)

```

저장 키: `notes:{customerId}` (배열)

### 첨부 파일

- PDF, 이미지, 텍스트
- 최대 10MB
- AI 분석 시 함께 전달

### AI 분석 출력 (마크다운, 6개 섹션)

1. 키워드
2. 한줄 니즈
3. 실제 니즈
4. 정제된 요구사항
5. 누락 항목
6. 리스크

→ AI 명세는 REQ-AI-001

### 자동 정보 추출

분석 후 별도 호출로 다음 항목을 JSON 추출:
- 도메인, 산업, 예산, 팀 구성, 의사결정권자, 다음 액션

추출 결과를 고객 정보 업데이트 제안 다이얼로그로 표시 (RM이 적용/거부).

## REQ-PROPOSAL-001 제안서 데이터 모델

```javascript
proposal = {
  recipient: string,
  rm_name: string,
  project_overview: string,    // RFP 관점
  decision_maker: string,
  collaborators: string,
  ai_config: {
    token_cap: 1000000          // 고정값 (POL-PRICING-002)
  },
  monitoring_subscriptions: [], // POL-MONITOR-001 (확장)
  plans: [
    {
      name: string,             // "A안" | "B안"
      variants: [               // 1~2개
        {
          mode: "ai_h" | "subscription",
          label: string,        // 선택적 부제
          scope_of_work: [],    // 업무 범위
          total_months: number, // 총 투입 기간
          monthly_hours: {      // 월별 투입 시간 오버라이드
            [memberId]: { "1": 40, "2": 30 }
          },
          members: [
            {
              id: string,
              name: string,
              required_skills: string,
              role_key: string,    // 작업자관리 직무명 또는 "__custom"
              role_custom: string, // role_key가 "__custom"일 때만
              grade: string,
              category: string,    // dev/plan/qa/publisher/rm/other
              contract_type: "hourly" | "monthly_fixed" | "short_term",
              weekly_hours: number,
              max_hours: number,
              rate: number,
              monthly_amount: number,
              duration: string,
              total_amount: number,
              is_harness: bool     // 하네스 엔지니어 여부 (기본 false). POL-PRICING-006
            }
          ]
        }
      ]
    }
  ]
}
```

저장: `customer.proposal_data` + `customer.proposal_history` (배열)

## REQ-PROPOSAL-002 제안서 생성 플로우 (3단계 다이얼로그)

**STEP 1**: 제안 구조 (단일안 / A안+B안)
**STEP 2**: A안 구성 선택 (체크박스, AI+H / 구독작업자 중 1~2개)
**STEP 3**: B안 구성 선택 (A/B 선택 시만, 동일 규칙)

기본값: 모두 `구독작업자`. 체크박스 최소 1개 강제.

다이얼로그 종료 → AI 생성 → 시간 검토 모달 → 확정.

## REQ-PROPOSAL-003 시간 검토 모달

AI 제안 직후 **확정 직전** 단계. 단일안/A·B안 모두 동일하게 작동.

### 구조

- 플랜(A/B) × variant(1안/2안) 탭으로 독립 검토 — 각 탭에서 멤버 카드 리스트 표시
- 총 투입 기간 (개월) 박스 (브랜드 강조). AI+H일 경우 `× 0.7` 적용된 실 기간을 우측에 함께 표시
- 멤버 카드 리스트 아래 `+ 포지션 추가` 버튼

### 멤버 행 편집 항목 (모두 인라인 수정 가능)

| 항목 | 입력 |
|---|---|
| 직무 | 드롭다운 (rates 기반) + `+ 직접 입력` 옵션 |
| 등급 | 드롭다운 (선택한 직무에 매칭되는 등급), 변경 시 시급 자동 갱신 |
| 시급 | 수동 오버라이드 가능 |
| 주 투입 시간 | number |
| 주 최대 시간 | number (신규 컬럼) |
| 계약 형태 | hourly / monthly_fixed / short_term |
| 하네스 토글 | **AI+H variant에서만 노출** (POL-PRICING-006) |
| 월별 시간 | months > 1 + hourly일 때 |
| 삭제 | 행 단위 |

### 하네스 멤버 시각

- 카드 좌측 3px solid `#9B87C9` 인디케이터
- 이름 옆 `🔧 하네스` 배지

→ 색상·배지 스펙은 DS-COLOR-002 참조

### 하단 요약

- **월 공급가** (가장 강조, 22px / 800) — POL-PRICING-002의 variant 합계 규칙으로 계산
- 총 기간 합계
- AI+H일 때만 토큰 운영비 안내 (POL-PRICING-003)

### 푸터 버튼

| 버튼 | 동작 |
|---|---|
| **확정하고 견적서 생성** | 현재 편집된 proposal로 확정. `proposal_history`에 새 엔트리 추가 후 미리보기 진입 |
| **AI 제안으로 다시 보내기** | REQ-PROPOSAL-008 플로우 시작 — 사유 입력 다이얼로그 → AI 재호출 → 다시 시간 검토 모달 |
| 취소 | 편집 폐기, 모달 닫음 |

### 호출 시점

REQ-PROPOSAL-002 다이얼로그 종료 후 AI 생성 완료 시, 또는 REQ-PROPOSAL-008로 재진입 시.

## REQ-PROPOSAL-008 AI 제안 재요청 플로우

REQ-PROPOSAL-003 시간 검토 모달의 "**AI 제안으로 다시 보내기**" 버튼이 트리거.

### 플로우

1. 클릭 → 사유 입력 다이얼로그 (`RevisionReasonDialog`)
   - textarea 1개, **최소 5자, 필수 입력**
   - placeholder 예시: "PM 비중을 줄이고 시니어 개발자 1명을 추가해주세요"
   - 푸터: [취소] [AI에 다시 요청]
2. 제출 → AI 호출 (`reviseFromReview(reason, editedProposal)`)
   - 시스템 프롬프트에 다음을 추가:
     - "사용자가 직접 수정한 결과(editedProposal JSON)"
     - "사용자 사유(reason)"
     - "이 방향을 반영해서 다시 제안하라"
   - 모델/구조 규칙은 REQ-AI-004 동일 (모드/variant 개수 강제 유지)
3. 응답 파싱 → 다시 `pendingProposal`로 세팅 → 시간 검토 모달 재진입 (반복 가능)
4. 사유는 `proposal_revision_patterns`에 누적 → REQ-AI-005

### proposal_history 처리

재요청으로 받은 새 제안은 시간 검토 모달에서 **확정**되어야만 `proposal_history`에 추가된다. 모달 안에서만 회차를 돌리는 동안은 `proposal_history`에 누적되지 않음.

## REQ-AI-005 수정 패턴 학습 시스템

REQ-PROPOSAL-008의 사유 입력을 누적 학습해 차후 제안 품질을 높인다.

### 저장소

- 키: `proposal_revision_patterns` (전역, REQ-DATAMODEL-001)
- 스키마:
  ```javascript
  pattern = {
    id: string,
    pattern: string,        // 사유 본문
    count: number,          // 누적 횟수
    active: bool,           // AI 프롬프트 주입 여부 (기본 true)
    created_at: ISO,
    updated_at: ISO
  }
  ```

### 자동 동작

1. "AI 제안으로 다시 보내기" 시 입력된 사유로 패턴 후보 생성.
2. AI 유사도 판정 (별도 호출): 모델 동일, 짧은 프롬프트로 *기존 active 패턴 각각과* "이 두 사유가 같은 의미인가? Y/N" 비교.
   - Y가 하나라도 있으면 → 첫 매칭 패턴의 `count++` & `updated_at = now`
   - 모두 N이면 → 새 패턴으로 추가 (`active=true`, `count=1`)
3. 추가 후 총합이 **20개 초과**면 자동 정리:
   - 가장 오래된 `active=false` 패턴부터 삭제
   - 모두 `active=true`면 가장 오래된 것 삭제

### AI 시스템 프롬프트 주입

`REQ-AI-003` 신규 생성과 `REQ-PROPOSAL-008` 재제안 호출 시:
- `proposal_revision_patterns`에서 `active=true` 항목들을 `count` 내림차순 정렬
- 0개면 섹션 생략
- 1개 이상이면 시스템 프롬프트 끝에 다음 추가:

  ```
  ## RM 피드백 패턴 (이전 제안에서 자주 지적된 사항, 우선 반영)
  1. {pattern} (count회)
  2. ...
  ```

### 관리 UI

설정 > AI 학습 패턴 탭 (REQ-SETTINGS-001). 컴포넌트: `RevisionPatternsPanel`.
